# Geometry Store

`GeometryStore` is the central geometry state manager. It holds all shapes, syncs to the DCEL, and integrates with history for undo/redo.

## What It Owns

- **All geometries**: stored in `geometryById: Map<Id, Geometry>`. Use `listWithComponent`/`listWithComponents` to query by component.
- **Working (transient) shapes**: `workingPolygon`, `workingRectangle`, `workingEllipse`, `workingConstraints` -- these represent shapes being actively drawn, NOT persisted
- **DCEL index**: `dcelIndex: DCELShapeIndex` for planar subdivision queries

All shapes use ID prefixes: `ply_` (polygon), `rct_` (rectangle), `elp_` (ellipse), `cns_` (constraint). IDs are generated by `HistoryManager.generateStableId()` which uses UUIDs plus a monotonic counter to prevent collisions across load/save cycles.

## Unified Component System

All geometry data is stored via a single `GeometryComponent<D>` (`src/lib/entity/components/GeometryComponent.ts`). Each geometry entity has exactly one geometry component whose data is a discriminated union:

```ts
type GeometryData = PolygonData | RectangleData | EllipseData;
```

Each `*Data` type has a `type` discriminator:

- `PolygonData` — `{ type: 'polygon', points, closed, openAtIndex, lastFillColor? }`
- `RectangleData` — `{ type: 'rectangle', upperLeft, lowerRight }`
- `EllipseData` — `{ type: 'ellipse', center, radiusX, radiusY }`

Read geometry data with `GeometryComponent.get(entity)`, which returns a `GeometryData`. Use `switch (data.type)` to discriminate. Update with `GeometryComponent.update(entity, partial)`.

The per-shape component types (`PolygonComponent`, `RectangleComponent`, `EllipseComponent`) are deprecated and used only internally by the old component definition files.

## CRUD Methods

All operations are unified by component detection rather than shape-specific methods:

| Operation                   | Records history           | No history recording                               |
| --------------------------- | ------------------------- | -------------------------------------------------- |
| Insert                      | `add(idPrefix, template)` | `addDirect(geometry)`                              |
| Update                      | `updateById(id, fn)`      | `updateByIdDirect(id, fn)`                         |
| Update (narrowed component) | --                        | `updateByIdWithComponentDirect(id, component, fn)` |
| Delete                      | `delete(id)`              | `deleteDirect(id)`                                 |

The history-recording methods call Direct internally, then record the operation to HistoryManager. HistoryManager uses only Direct methods during undo/redo replay to avoid infinite recording loops.

Direct methods emit `geometryAdded`/`geometryUpdated`/`geometryDeleted` events so the UI stays responsive during undo/redo. They just don't loop back into history.

## Query Methods

| Method                                    | Returns                                                                     |
| ----------------------------------------- | --------------------------------------------------------------------------- |
| `listWithComponent(component)`            | `Array<Entity<C>>` — all geometries with that component                     |
| `listWithComponents(a, b, c?, d?)`        | `Array<Entity<A & B & (C & D)?>>` — all geometries with multiple components |
| `getById(id)`                             | `Entity \| null`                                                            |
| `getByIdWithComponent(id, component)`     | `Entity<C> \| null` — narrowed by component                                 |
| `getByIdWithComponents(id, a, b, c?, d?)` | `Entity<A & B & ...> \| null` — narrowed by multiple components             |

To get all geometry entities: `store.listWithComponent(GeometryComponent)` returns `Array<Geometry>`.

## DCEL Sync

- **Add/delete**: DCEL is updated immediately (synchronously) via `dcelIndex.addGeometry`/`dcelIndex.removeGeometry`.
- **Update** (during drag): DCEL sync is **debounced per shape ID by 200ms** (via lodash) via `_syncDcelUpdate`. This keeps the hot path fast during rapid mouse movement.
- **Immediate flag**: The `reconstrain()` method bypasses debounce with `immediate=true` so constraint solving results appear instantly.

## Events

Emitted on `EventEmitter`:

| Event               | Payload               |
| ------------------- | --------------------- |
| `geometryAdded`     | `geometry: Entity`    |
| `geometryUpdated`   | `geometry: Entity`    |
| `geometryDeleted`   | `geometryId: Id`      |
| `workingXxxChanged` | working shape or null |

Subscribe to `geometryAdded`/`geometryUpdated`/`geometryDeleted` and re-query with `listWithComponent` as needed.

## Constraint Endpoint Resolution

`resolveConstraintEndpoint(endpoint)` handles four endpoint types:

- `"point"` -- returns the literal `SheetPosition`
- `"locked-rectangle"` -- looks up geometry, computes corner via `GeometryComponent.keyPoints`
- `"locked-ellipse"` -- looks up geometry, computes key point via `GeometryComponent.keyPoints`
- `"locked-polygon"` -- looks up geometry, returns the indexed segment's point

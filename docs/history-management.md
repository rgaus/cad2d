# History Management

`HistoryManager` manages undo/redo with a stack-based approach plus transaction support for grouping atomic operations.

## Undo/Redo Stacks

Two LIFO stacks: `undoStack` and `redoStack` of `UndoEntry` discriminated union types. There are ~31 entry types covering every undoable operation (polygon/rect/ellipse/conversion/constraint CRUD, render order, fill color, point insertion, etc.).

- `undo()`: pop from undoStack, apply reverse, push to redoStack
- `redo()`: pop from redoStack, apply forward, push to undoStack
- Both emit `stacksChange` event (used by UndoAction/RedoAction to enable/disable buttons)

## Transactions

`recordTransaction(purpose, scopeFn)` groups multiple individual undo entries into a single atomic unit:

1. Buffers all entries recorded during `scopeFn()` into `activeTransaction`
2. Wraps them in a `TransactionEntity` with a human-readable purpose string (e.g., `'boolean-union'`, `'reconstrain'`)
3. Undo replays entries in **reverse** order; redo replays in **forward** order
4. Supports nesting (transactions within transactions)

This is used for:
- Boolean operations (`'boolean-union'`, `'boolean-difference'`, `'boolean-intersection'`)
- Constrained shape creation (`'create-rectangle-with-constraints'`)
- Constraint solving (`'reconstrain'`)

## Direct Method Pattern (Critical)

The `applyForward`/`applyReverse` switch statements use only **Direct** methods on GeometryStore (`addPolygonDirect`, `deletePolygonDirect`, `updatePolygonDirect`, etc.). Direct methods:
- Mutate geometry state
- Sync DCEL
- Emit change events
- Do NOT record to history

This prevents infinite loops (HistoryManager -> GeometryStore -> HistoryManager -> ...).

## State Serialization

Full undo/redo stacks are serialized into the SVG file's magic state comment (`<!-- cad2d-state:... -->`). The `stableIdCounter` is also preserved to prevent ID collisions after load.

## Stable ID Generation

All shape IDs are generated by `HistoryManager.generateStableId(prefix)` which uses `uuid` plus a monotonic counter. This counter is serialized so that new shapes created after loading a file don't collide with restored shapes.
